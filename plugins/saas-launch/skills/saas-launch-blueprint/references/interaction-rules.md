# Interaction Rules

Canonical home for the SaaS Launch Blueprint's **process rules** — how you work and how you talk to the founder, as opposed to what the product/business constraints are (those live in `shared-context.md`) or what the engineering/design/video standards are (those live in `tech-design-video.md`). Every phase skill in this workflow (Phase 1 ideation, Phase 2 PRD, Phase 3 prototype, Phase 3.5 handoff) reads this file at its start and operates under these rules for its entire run. Nothing here is phase-specific; it applies EVERYWHERE across the workflow. Throughout this file, first-person pronouns (I / ME / MY) refer to the USER — the founder; "you" is Claude.

## Delegation Model (this session, Phases 1–3.5)
You, the executing agent, are the strategist. Reserve your own context and effort for the work only you can do well: planning, brainstorming, architectural design, interviewing the user, writing the PRD, and writing the handoff prompt. Do NOT do heavy legwork inline:
- Web research — market sizing, competitor scans, pricing references, free-tier limits, domain availability checks, legal/regulatory references — fans out to subagents: Sonnet for research requiring judgment, Haiku/Explore for simple lookups. Launch independent research tasks in PARALLEL and synthesize their findings yourself.
- Simple, trivial, mechanical tasks — file conversions, boilerplate, renames, packaging, render invocations — delegate to Haiku-class helpers whenever possible.
- Keep delegation scoped: subagents fetch and summarize; YOU decide. Never outsource the ideation scoring, PRD judgments, design direction, or anything the user will be interviewed about — those are the deliverables this skill exists for.
Why this matters: inline research bloats your context and measurably degrades the strategic outputs (idea quality, PRD coherence, prompt precision) that are your actual job — and parallel fan-out is faster anyway.

## Known Failure Modes (each observed in real runs — do not repeat them)
1. **Widget with no context.** A selection widget appeared listing bare product names; the user had nothing to base a decision on. The analysis existed only as narration between tool calls, which the UI summarized away. This is why AskUserQuestion rules 1–3 demand the standalone write-up land visibly BEFORE the widget.
2. **Deliverables never shown.** A prototype phase "finished" with files written to disk and described in prose — nothing clickable, nothing rendered — and the approval gate fired anyway. This is why the DELIVERABLE PRESENTATION rule requires files in front of the user before any gate question.
3. **HTML instead of video.** The pitch "clip" shipped as an HTML composition with no rendered MP4, quietly reframed as if that were the deliverable. This is why the Video tooling constraint (see `tech-design-video.md`) counts only a rendered .mp4 — and why the clip is opt-in (Phase 3 STEP 3) rather than silently attempted.
4. **Stack drift.** Builds shipped without Tailwind v4 + shadcn/ui, or with hand-rolled design systems, on a PRD that had locked the template's UI stack. Drift means shipping anything other than what the PRD's `Stack source` line locked — on the template path, the UI stack is non-negotiable in production and the design system must derive from shadcn/ui via the deterministic scaffold; on the custom path, the founder's locked CUSTOM STACK SPEC governs with the same force. Neither path may silently substitute a different stack from what the PRD recorded.

These are not hypotheticals; each one cost the user a session. Whenever speed and one of these rules pull in opposite directions, the rule wins.

## Presentation Environment (detect once, reuse all run)

At phase start, determine which client you are running in and carry that determination for the entire run. This ONE determination governs both how you present option comparisons (AskUserQuestion rule 6) and how you present deliverables (Deliverable Presentation rule):

- **Claude Cowork** — supports the user-message tool, inline HTML widgets, and file-presentation cards. Use the rendered-widget path.
- **Claude Code / the TUI** — assistant text renders as GitHub-flavored markdown (tables, headers, bold, code blocks all display well), but there is NO user-message tool, NO inline HTML widget, and NO clickable file card. AskUserQuestion IS available and additionally supports a per-option `preview` field (markdown, single-select questions only). Use the markdown-native path.
- **Any other client** — assume TUI-class (markdown-native, no widgets) unless you can positively confirm widget support.

Neither path is a downgrade. Cowork gets rendered widgets; the TUI gets rich markdown authored for it — a comparison table reads at least as clearly as a scorecard widget. Each is the FIRST-CLASS presentation for its environment. Never present the TUI path as "widget unsupported, skipped" — you replace the widget with a better-for-the-TUI format, you do not drop it.

## AskUserQuestion rules (apply EVERYWHERE, HARD REQUIREMENT)
1. ALWAYS deliver a full write-up of the options BEFORE calling AskUserQuestion — each option's name, a clear explanation of what it is, key details, and trade-offs. Never call AskUserQuestion cold.
2. The write-up must reach the user as a COMPLETE, VISIBLE message before the widget appears. In Cowork, push it via the user-message tool (it renders verbatim mid-run), then call AskUserQuestion. In clients without that tool — Claude Code / the TUI, and any non-Cowork client — emit the write-up as the final uninterrupted text block immediately preceding the AskUserQuestion call (there is no user-message tool there, so that text block is the only way the full context reaches the founder before the prompt). What is forbidden is narration scattered between unrelated tool calls — that text gets summarized away and the user sees a context-free widget.
3. SELF-CHECK before every AskUserQuestion call, on BOTH counts: (a) has the full write-up for these exact options visibly reached the user? and (b) does EVERY option I am about to pass carry a populated per-option `description` — 1–2 concrete sentences, never a bare label (rule 4)? If either check fails, STOP and fix it before calling.
4. **PER-OPTION DESCRIPTION IS MANDATORY — every AskUserQuestion call, no exceptions (selection widgets AND approval gates).** Every option object you pass to AskUserQuestion MUST populate the tool's per-option `description` field with 1–2 concrete sentences: what the option IS / what choosing it does, plus its single standout trade-off. A bare label — empty, omitted, or whitespace `description` — is FORBIDDEN and counts as a failed call; the founder must NEVER see an option with nothing under it. The recommended option's `description` states plainly WHY it is recommended. This is not a restatement of the write-up: the write-up holds full context, and the `description` is the per-option gist that must stand on its own inside the widget. Widget option names must match the write-up's names exactly. This requirement holds identically for the three standing approval-gate options (see APPROVAL GATES).
5. When multiple decisions happen back to back (segment → scope → product idea), EACH decision gets its own write-up and its own widget — never chain widgets without the intervening write-ups.
6. **COMPARISON FORMAT (environment-aware; SUPPLEMENTS the write-up, never replaces it).** Every major selection — customer segment, geographic scope, product idea, stack source, tech-stack deviation — MUST carry a structured side-by-side comparison in addition to the prose write-up required by rules 1–3. Render it in the form native to the detected environment (see the Presentation Environment section):
   - **Cowork:** render an inline visual comparison scorecard widget alongside the write-up — the existing behavior, unchanged.
   - **Claude Code / the TUI (and any non-widget client):** embed a rich markdown comparison/scorecard TABLE directly inside the standalone write-up — criteria as columns, options as rows, an explicit score in every cell (filled/empty dots like ●●●○○ or a 1–5, plus a short word where a number needs nuance), and a final **Verdict** row or column naming the recommendation. This table IS the scorecard in the TUI: a genuinely readable, first-class format built for markdown, not a fallback for a missing widget. When the decision is single-select, ALSO fill each AskUserQuestion option's `preview` field with a compact per-option snippet (its standout scores + the one-line trade-off) so the comparison reappears at the moment of choosing — the preview supplements the write-up table, it never replaces it.
   In every environment a reader seeing ONLY the text must still fully understand every option, and the write-up covers ALL options even when the widget/preview surface is capped (rule 7).
7. OPTION CAP: the widget holds at most 4 options. When a decision has more, show the top 4 by the decision's own ranking (segments: listed order; ideas: total score, recommendation first) and state in the write-up that the remaining options are reachable via "Other". The write-up always covers ALL options, capped widget or not.

## APPROVAL GATES (hard rule)
Every phase ENDS with an explicit approval prompt via AskUserQuestion carrying THREE standing options: "Approve — proceed", "Request changes" (iterate in place), and "Go back — revisit an earlier decision" (reopens the named earlier phase; re-approving an earlier gate invalidates all downstream deliverables, which must be revised or regenerated). Each of these three options MUST populate its per-option `description` field per AskUserQuestion rule 4 — Approve: what proceeding triggers next; Request changes: that it iterates the current phase in place without advancing; Go back: which earlier decision it reopens and that downstream deliverables are then invalidated. Bare gate labels are forbidden. NO phase begins until the previous gate is approved. Gates exist so a wrong early guess — segment, idea, pricing — never fossilizes into the expensive phases.

Each phase skill's own gate (GATE 1, GATE 2, GATE 3 — Phase 3.5 forks: on the hand-off branch it has no gate of its own (the final "Accept delivery" gate is encoded in the BUILD-HANDOFF.md prompt and runs in the separate build session); on the build-now branch Phase 3.5 ends with its own GATE 4 — "Accept delivery" / "Request changes" — run in this same session) adds its own phase-specific option labels on top of these three standing options — see that phase skill for the exact wording. The mechanics below are the same at every gate regardless of phase:
- **"Approve — proceed"** hands control back to the orchestrator, which immediately invokes the next phase's skill. A phase skill finishing is NOT the end of the workflow — it only pauses at the gate for the user's decision.
- **"Request changes"** keeps control inside the current phase skill: iterate in place and re-present the same gate. The orchestrator does not advance.
- **"Go back — revisit an earlier decision"** returns control to the orchestrator, which re-invokes the named earlier phase's skill; that phase's downstream deliverables are treated as invalidated and must be revised or regenerated once the earlier decision changes.

## DELIVERABLE PRESENTATION (hard rule)
A phase is NOT complete when its deliverables are merely written to disk and described in text — they must be put IN FRONT of the founder before the gate question appears:
- In Claude Cowork: present every deliverable file with the file-presentation tool (clickable cards), AND render HTML deliverables inline as widgets. For LARGE sets (e.g., a storyboard), cards for ALL files but inline rendering for a representative subset only (3–5 key frames or a thumbnail index) — duplicating every byte inline defeats the Delegation Model's context economy.
- In Claude Code / the TUI / Claude Desktop (per the Presentation Environment section — no clickable cards, no inline HTML render): present every deliverable as its ABSOLUTE path in a single copyable block, each path paired with a one-line open command (`wslview` in WSL, `start` on Windows, `open` on macOS), or a clickable `file://` link where the client supports it. HTML deliverables (the prototype, the Launch Dossier) open in the founder's browser via that command — say so plainly rather than implying an inline preview. This copyable-paths block is the first-class deliverable presentation for the TUI, not a degraded one.
- NEVER call the gate AskUserQuestion before the deliverables under review have been presented this way.
