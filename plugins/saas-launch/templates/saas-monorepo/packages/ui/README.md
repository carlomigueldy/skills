# @{{PRODUCT_SLUG}}/ui

Shared design-system package for {{PRODUCT_NAME}}.

## The one rule

**Components come from shadcn/ui, customized via `src/tokens.css`. Never
hand-roll a UI primitive in this package (or in any app).**

- `src/tokens.css` is the design system: Tailwind v4 `@theme` + CSS variables
  for color, radius, and font. Edit values here, not in components.
- Components are generated by the shadcn/ui CLI directly into
  `src/components/`, wired to read from these tokens. They are regular,
  editable source files once generated — but they *originate* from the CLI,
  not from scratch, so upgrades stay diffable via `shadcn diff`.
- `src/index.ts` re-exports generated components plus the shared `cn()`
  classname helper.

## Adding or upgrading a component

Run from this package's directory (`packages/ui`), after `pnpm install`:

```bash
pnpm dlx shadcn@latest add <component-name>
```

See `../../SCAFFOLD.md` for the exact init flags and the full baseline
component list used when a product is first scaffolded.

## Why nothing is vendored here

This template ships tokens and the `cn` helper only — no generated `.tsx`
component output and no lockfile. Component output is generated fresh per
product so it can pick up the latest shadcn/ui source and stay aligned with
that product's own token values, instead of drifting from a stale copy
baked into this template. See `../../SCAFFOLD.md` → "What is intentionally
NOT vendored" for the full rationale.
