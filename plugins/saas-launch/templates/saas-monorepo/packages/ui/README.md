# @{{PRODUCT_SLUG}}/ui

Shared design-system package for {{PRODUCT_NAME}}.

## The one rule

**Components come from shadcn/ui, customized via `src/tokens.css`. Never
hand-roll a UI primitive in this package (or in any app).**

- `src/tokens.css` is the design system: Tailwind v4 `@theme` + CSS variables
  for color, radius, and font. Edit values here, not in components.
- Components are **vendored** — pre-generated by the shadcn/ui CLI (style
  `new-york-v4`) directly into `src/components/ui/`, wired to read from
  these tokens. They are regular, editable source files, checked into this
  template — not regenerated at scaffold time.
- `src/index.ts` re-exports the `cn()` classname helper plus all vendored
  component modules.

## What's vendored

14 baseline components ship in `src/components/ui/`:

`badge` `button` `card` `dialog` `dropdown-menu` `form` `input` `label`
`select` `sheet` `skeleton` `sonner` `table` `tabs`

Plus `src/lib/utils.ts` (the `cn()` helper — canonical source, re-exported
from `src/index.ts`) and `components.json` (style `new-york-v4`,
`cssVariables: true`, `baseColor: neutral`).

Component source here uses **relative imports**
(`../../lib/utils`, `./button`), not the shadcn-standard `@/...` alias —
`components.json`'s `@/...` aliases exist for the CLI's own file-placement
logic (and for extending app-local component dirs), but a nested workspace
package's own internals need imports that resolve when bundled by a
consuming app, which a bare `@/*` alias inside `packages/ui` cannot
guarantee across every app's bundler config. Keep this convention when
adding components by hand (see below).

## Adding or upgrading a component

Use the **`shadcn` agent skill bundled with every scaffolded project** at
`.claude/skills/shadcn/` — it already knows this project's aliases, the
`new-york-v4` style, and the Tailwind v4 setup from `components.json`.
Invoke it (or just ask for the component by name) rather than running the
CLI from memory.

Equivalent by hand, run from this package's directory (`packages/ui`),
after `pnpm install`:

```bash
pnpm dlx shadcn@latest add <component-name>
```

Then, before committing:

1. Convert the newly-added file's `@/...` imports to the relative-import
   convention used by every other file in `src/components/ui/` (see
   "What's vendored" above) — the CLI writes `@/...` by default, which
   won't resolve inside this package once it's bundled by a consuming app.
2. Re-export the new component from `src/index.ts` alongside the existing
   14.

See `../../SCAFFOLD.md` §3.5 for the full context on why this is optional
and additive, never a required scaffold-time step.

## Why this is vendored (and wasn't before)

Earlier revisions of this template generated component output fresh at
scaffold time via `shadcn init` + `shadcn add`, on the theory that each
product would then track the latest shadcn/ui source. In practice that
made every first-hour scaffold depend on network access, CLI version
drift, and preset changes upstream — a deterministic template with a
non-deterministic first step. Vendoring trades that for an occasional
manual re-vendor pass in this template when there's an upstream shadcn/ui
change worth pulling in. Either way, components stay upgradeable via
`shadcn diff` / `shadcn add --overwrite` (see above) — vendoring only
changes when the CLI runs, not whether it stays available.
